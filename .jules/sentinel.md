## 2025-04-20 - HTML Injection in Contact Form Emails
**Vulnerability:** The contact form endpoint (`/api/contact/route.ts`) takes user input for `name`, `email`, `subject`, and `message` and interpolates it directly into the HTML body of an email sent via Resend. The endpoint does not sanitize this input (except for a `.replace(/\n/g, "<br>")` on the message field), which allows an attacker to inject arbitrary HTML into the support email, potentially leading to phishing, misleading content, or rendering exploits in the email client.
**Learning:** This existed because while `zod` provides validation, it does not provide HTML sanitization by default. Developers often assume internal email recipients are safe from XSS, but HTML injection in support emails is a vector for social engineering or email client exploits. Also, there's a memory mentioning "The base `sendEmail` utility (`src/lib/email.ts`) auto-sanitizes headers... However, it does not auto-sanitize HTML body inputs. Callers must manually sanitize user inputs using `escapeHtml` (from `src/lib/html-sanitizer.ts`) before passing them into HTML templates to prevent injection vulnerabilities." and we found `src/lib/html-sanitizer.ts` did not exist!
**Prevention:** Always escape user input before interpolating it into HTML contexts, even for internal tooling or emails. Create a reusable `escapeHtml` utility and apply it systematically to all email templates.
